API REFERENCE

RESTful API for KAVACH Platform Integration

Authentication

The KAVACH API uses API keys for authentication. Include your API key in the Authorization header of all requests. API keys can be generated and managed in the Settings > API Keys section of the platform.

# Example: Authentication Header curl -X GET https://api.kavach.io/v1/incidents \ -H "Authorization: Bearer kvch_live_aBcD3FgH1JkLmN0pQrStUvWxYz" \ -H "Content-Type: application/json"
Security Best Practices
Never expose API keys in client-side code or public repositories. Rotate keys regularly and immediately revoke any compromised keys. Use environment variables or secure key management systems to store API credentials.

Rate Limits

Endpoint Type Rate Limit Window
Event Ingestion 10,000 requests Per minute
Query & Search 1,000 requests Per minute
Incident Management 500 requests Per minute
Report Generation 100 requests Per hour

Rate limit information is included in response headers: X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset.

Event Ingestion API

POST /v1/events/ingest

Ingest security events into KAVACH for processing and correlation. Supports batch ingestion of up to 1,000 events per request.

Request Body Parameters
events array required

Array of event objects to ingest

timestamp string (ISO 8601) required

Event timestamp in ISO 8601 format

source string required

Data source identifier (e.g., "firewall", "endpoint-edr")

event_type string required

Type of event (e.g., "network", "process", "authentication")

severity string

Event severity: "critical", "high", "medium", "low", "info"

metadata object

Additional event metadata as key-value pairs

// Example Request { "events": [ { "timestamp": "2025-01-02T14:32:18Z", "source": "palo-alto-firewall", "event_type": "network", "severity": "high", "metadata": { "src_ip": "192.168.1.100", "dst_ip": "185.220.101.42", "dst_port": 443, "action": "allow" } } ] }
Responses

200 Events successfully ingested

400 Invalid request format or missing required fields

401 Authentication failed

GET /v1/events/search

Search and query ingested events using filters, time ranges, and field matching.

Query Parameters
query string

Search query string (supports field:value syntax)

start_time string (ISO 8601)

Start of time range for search

end_time string (ISO 8601)

End of time range for search

limit integer

Maximum number of results (default: 100, max: 1000)

offset integer

Pagination offset

# Example: Search for failed login attempts GET /v1/events/search?query=event_type:authentication%20AND%20action:failed&limit=50

Incident Management API

GET /v1/incidents

Retrieve a list of detected incidents with optional filtering by severity, status, and time range.

// Example Response { "incidents": [ { "id": "inc_2025_001", "title": "Lateral Movement Detected", "severity": "critical", "status": "active", "created_at": "2025-01-02T14:15:00Z", "affected_assets": ["WS-042", "DB-SERVER-03"], "confidence_score": 0.96 } ], "total": 23, "page": 1 }
GET /v1/incidents/{incident_id}

Retrieve detailed information about a specific incident, including complete kill-chain reconstruction and timeline.

PUT /v1/incidents/{incident_id}

Update incident status, add analyst notes, or modify incident metadata.

// Example: Update incident status { "status": "resolved", "notes": "Threat contained and remediated. Affected systems rebuilt." }

Report Generation API

POST /v1/reports/generate

Generate a forensic report for a specific incident or time range. Reports are generated asynchronously and can be retrieved using the report ID.

Request Body Parameters
incident_id string

Incident ID to generate report for

format string

Report format: "pdf", "json", "html" (default: "pdf")

template string

Report template: "forensic", "executive", "technical", "compliance"

// Example Response { "report_id": "rpt_abc123", "status": "generating", "estimated_completion": "2025-01-02T14:45:00Z" }
GET /v1/reports/{report_id}

Retrieve a generated report by its ID. Returns report metadata and download URL when generation is complete.

Webhooks

Configure webhooks to receive real-time notifications when incidents are detected or status changes occur. KAVACH will POST JSON payloads to your configured endpoint.

// Webhook Payload Example { "event": "incident.created", "timestamp": "2025-01-02T14:15:00Z", "data": { "incident_id": "inc_2025_001", "severity": "critical", "title": "Lateral Movement Detected" } }

Webhook endpoints must respond with HTTP 200 within 10 seconds. Failed deliveries will be retried with exponential backoff.
